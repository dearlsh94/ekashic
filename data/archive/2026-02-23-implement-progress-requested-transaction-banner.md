# Implement Progress Requested Transaction Banner

## Objective
ProgressRequestedTransactionBanner 컴포넌트의 TODO를 완전히 구현하여 결제 처리 중인 거래에 대한 사용자 알림 배너를 표시하는 기능을 완성한다.

## Problem
- ProgressRequestedTransactionBanner 컴포넌트에 "TODO 단말기명, 결제수단 이름, 가격 표시, 다국어 추가" 주석만 있고 실제 구현이 없었음
- REQUESTED 상태인 거래들에 대해 사용자에게 진행 상황을 알려주는 UI가 필요했음
- 단말기 이름, 결제수단 정보, 가격 등을 포함한 구체적인 정보 표시가 요구됨

## Decisions

1. **기존 인프라 활용**: PaymentSettings와 Terminal 관련 도메인이 이미 완성되어 있어 새로운 도메인 모델 생성 없이 기존 것을 활용하기로 결정

2. **다중 배너 지원**: 여러 REQUESTED 거래가 있을 때 각각을 별도 배너로 표시하여 사용자가 모든 진행 중인 거래를 확인할 수 있도록 결정

3. **배너 전용 결제수단 포맷팅**: 기존 `useGetTransactionPaymentTypeTitle` 대신 배너만을 위한 단순한 포맷팅 로직을 구현하여 "[결제 단말기]" prefix 제거하기로 결정

4. **할부 정보 포함**: 카드와 간편결제 모두에서 할부 개월 정보를 표시하여 더 구체적인 결제 정보를 제공하기로 결정

5. **최신순 정렬**: 여러 REQUESTED 거래가 있을 때 `lastUpdatedDateTime` 기준으로 최신 거래부터 표시하기로 결정

## Key Learnings

1. **코드베이스 재사용**: 새로운 기능 구현 전에 기존 인프라를 충분히 조사하면 개발 시간을 대폭 단축할 수 있음. Terminal 관련 기능이 이미 완성되어 있어 구현 복잡도가 크게 줄어듦

2. **사용자 중심 설계**: 단일 배너가 아닌 다중 배너 지원으로 사용자가 모든 진행 중인 거래를 명확히 파악할 수 있게 함

3. **컴포넌트별 책임 분리**: 배너 컴포넌트만의 간단한 결제수단 포맷팅을 구현하여 다른 컴포넌트와의 의존성을 최소화

4. **타입 안전성**: Transaction 도메인의 타입을 적극 활용하여 런타임 오류 가능성을 원천 차단

## Impact

### Positive Impact
- **사용자 경험 개선**: 결제 처리 중인 거래에 대한 명확한 시각적 피드백 제공
- **정보 투명성**: 단말기명, 결제수단, 금액 등 구체적 정보로 사용자 불안감 해소
- **다중 거래 지원**: 여러 거래 동시 처리 상황에서도 각각의 진행 상황을 명확히 표시
- **유지보수성**: 배너 전용 로직으로 다른 컴포넌트와의 결합도 최소화

### Technical Debt Reduction
- TODO 주석 제거로 기술 부채 해결
- 타입 안전성 확보로 런타임 오류 위험 감소
- 기존 도메인 모델 재사용으로 중복 코드 방지

### Business Value
- 결제 프로세스의 투명성 증대로 사용자 신뢰도 향상
- 명확한 진행 상황 표시로 고객 서비스 품질 개선
- 다중 거래 지원으로 복잡한 비즈니스 시나리오 대응 능력 강화
