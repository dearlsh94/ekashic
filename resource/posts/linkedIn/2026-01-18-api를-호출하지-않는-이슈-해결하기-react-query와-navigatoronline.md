API를 호출하지 않는 이슈 해결하기 (React Query와 navigator.onLine)

작년 저희 팀이 전달받은 이슈 리포트 중 원인을 발견하기까지 고생했던 문제가 있었습니다. 바로 특정 사용자의 PC 환경에서 접속 시, 일부 API가 아예 호출되지 않는 문제였습니다. 이로 인해 사용자는 일부 화면이나 기능을 정상적으로 사용할 수 없었습니다. 특이한 점은 다른 사용자들은 모두 정상 동작했고, 오직 특정 사용자 1명에게서만 발생했다는 것입니다.

단일 사용자 환경에서만 발생하는 이슈였기 때문에, 몇몇 가설 기반하에 단계적으로 문제의 원인을 찾아나갔습니다. 특정 계정에 설정된 데이터의 문제는 아닌지, 브라우저 환경의 문제는 아닌지, VPN이나 방화벽 혹은 외부 보안 프로그램의 문제는 아닌지 등에 대한 가설을 하나씩 세워가며 검토해 나갔습니다. 하지만 모두 문제의 원인이 아니었습니다. 이게 문제일 리가 없는데, 하면서도 지푸라기라도 잡는 심정으로 여러 가설을 검증해 나갔습니다.

그러다 호출되지 않는 API들이 모두 React Query를 사용하고 있다는 공통점을 찾았습니다. React Query는 데이터 페칭에 대한 자체적인 최적화 논리를 가지고 있고, 이로 인해 네트워크 탭에서도 확인할 수 없는 경우가 있습니다. 호출, 캐싱, 반환값 처리 등 어떤 부분에서의 문제인지를 파헤쳐갔습니다. 그러면서도 한 편으로 다른 가설들을 계속 세워가기도 했습니다.

그리고 마침내 React Query는 브라우저가 네트워크에 연결되어 있는지를 브라우저의 navigator.onLine 속성에 의존하고 있다는 것을 알았고, 이에 따라 오프라인 상태라면 네트워크 호출 요청을 하지 않는다는 것도 알 수 있었습니다.

개발 환경에서 이 가설을 검증했을 때 문의자 분의 상황과 동일한 증상이 재현되었으며, 실제 해당 사용자 PC 환경에서 onLine 속성 또한 false임을 확인할 수 있었습니다. 이제 드디어 문제 지점을 찾았습니다.

문제는 navigator.onLine의 판단 한계와 여기에 네트워크 연결 상태 판단을 의존한 React Query의 최적화 논리에 있었습니다. 문의자 분의 환경은 공유 오피스의 유선 네트워크를 사용하고 있었고, 외부 특정 도메인 접근이 제한되거나 불안정했습니다. 하지만 서비스 도메인과 API 서버는 접근할 수 있었습니다. 결과적으로 브라우저는 오프라인으로 판단했지만, 서비스는 정상 동작 가능한 모순적 상태가 되었습니다.

우선 문의자분의 네트워크 환경에 대해서 응대하고, React Query의 networkMode를 'always'로 설정하여 브라우저의 네트워크 판단과 무관하게 요청을 항상 시도하도록 수정하여 애플리케이션에서 API 성공과 실패를 확인할 수 있도록 변경한 핫픽스를 배포하며 이슈를 종료할 수 있었습니다.

브라우저의 onLine 속성에 네트워크 연결 상태를 의존하는 것은 위험합니다. 크롬 브라우저의 문제로 잘못 판단될 수도 있으며, 사용자의 특수한 네트워크 환경을 고려하지 못합니다. 그렇기 때문에 React Query도 이 의존성 문제를 해결한 것일테죠. (참고: https://lnkd.in/g3Cc68r4)

의존 관계에서 담보되는 위험성과 라이브러리가 제공해 주는 최적화의 양면성에 관한 생각을 다시 일깨워주는 에피소드라 지금까지 겪은 이슈들 중 꽤나 인상깊게 남아있습니다. 일반적 환경에 최적화 되어있다고 하여 모든 서비스 환경에서 안정적으로 사용할 수 있다는 섣부른 낙관 또한 조심합시다.
